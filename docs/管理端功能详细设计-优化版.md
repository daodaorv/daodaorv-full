## 4. 核心业务管理功能

### 4.1 用户管理功能

#### 4.1.1 用户列表管理

**业务场景**：

- **场景 1**：运营人员需要查找特定用户进行客服支持
- **场景 2**：财务人员需要导出用户数据进行对账分析
- **场景 3**：运营人员需要对异常用户进行批量处理
- **场景 4**：数据分析人员需要按不同维度统计用户数据

**功能描述**：

- 查看所有注册用户信息
- 多维度筛选和搜索用户
- 批量操作用户状态
- 导出用户数据报表

**用户使用流程**：

1. 运营人员登录管理端，进入用户管理模块
2. 根据业务需求选择筛选条件（用户状态、会员类型、注册时间等）
3. 如需查找特定用户，在搜索框输入用户名或手机号
4. 查看筛选结果列表，点击用户查看详细信息
5. 如需批量操作，勾选多个用户后选择批量操作（冻结/解冻/导出等）
6. 如需导出数据，选择导出格式和字段，生成报表文件

**功能价值**：

- **提升运营效率**：快速定位目标用户，减少查找时间
- **支持数据分析**：多维度筛选和导出功能，支持数据分析和决策
- **批量处理能力**：支持批量操作，提高处理效率
- **信息安全**：手机号脱敏显示，保护用户隐私

**筛选条件**：

- 用户状态：全部/正常/冻结/封禁
- 会员类型：普通用户/PLUS 会员/众筹车主
- 注册时间：今日/本周/本月/自定义时间段
- 资料状态：已提交/未提交/审核中/审核通过/审核拒绝

**列表字段**：

- 用户头像、昵称、手机号（脱敏）
- 会员类型、注册时间
- 资料状态（实名、驾照）
- 账户状态、最后登录时间
- 操作按钮：查看详情、编辑、状态管理

**业务规则**：

- 手机号脱敏显示：138\*\*\*\*8888
- 普通管理员只能查看，不能编辑
- 运营管理员可以编辑基本信息
- 超级管理员可以修改所有信息

#### 4.1.2 用户资料审核

**业务场景**：

- **场景 1**：用户提交实名认证资料，运营人员需要审核通过后才能使用租车服务
- **场景 2**：用户提交驾照认证资料，运营人员需要验证驾照有效性
- **场景 3**：用户资料审核被拒绝后重新提交，运营人员需要重新审核

**用户使用流程**：

1. 用户在客户端提交实名认证或驾照认证资料
2. 系统自动将待审核资料推送到管理端审核队列
3. 运营人员进入审核页面，查看用户提交的资料
4. 运营人员检查资料格式和清晰度，验证信息一致性
5. 审核通过：用户可正常使用相关功能
6. 审核拒绝：填写拒绝原因，用户收到通知后可重新提交

**实名资料审核**：

- **审核内容**：真实姓名、电话号码、身份证号码、身份证正反面照片
- **审核标准**：照片清晰、信息真实、格式正确、信息一致
- **审核结果**：通过或拒绝（需填写原因）

**驾照资料审核**：

- **审核内容**：驾驶证号码、驾驶证正副页照片、电话号码
- **审核标准**：照片清晰、号码格式正确、在有效期内、信息一致
- **审核结果**：通过或拒绝（需填写原因）

**功能价值**：

- **保障业务安全**：通过资料审核确保用户身份真实，降低业务风险
- **合规要求**：满足相关法规对身份认证的要求
- **用户体验**：资料审核通过后用户可享受完整服务

**业务规则**：

- 资料审核仅检查格式正确性，不进行在线验证
- 实际验证在线下交付时进行
- 审核拒绝需填写详细原因
- 用户可重新提交资料

#### 4.1.4 用户标签管理

**功能描述**：为用户添加业务标签，用于精准营销和权益管理。

**标签类型**：

- **系统标签**：PLUS 会员、众筹车主、平台购车用户、特殊推广用户、员工
- **行为标签**：高频租车、优质客户、投诉用户
- **自定义标签**：运营人员可自定义标签

**标签应用**：

- 推广分享收益计算
- 精准营销和消息推送
- 用户分群和数据分析
- 特殊权益和服务

**业务规则**：

- PLUS 会员标签：用户付费开通后自动添加
- **众筹车主标签**：
  - 添加时机：众筹项目正式配置车辆并由后台确认运营后自动添加
  - 添加条件：用户持有份额 > 0 AND 项目状态 = 'operating'
  - 添加对象：所有持有该项目份额的用户（在项目启动运营时批量添加）
  - 份额转让后：新买家获得份额时，如果项目运营中，立即添加标签
  - 移除时机 1：用户卖出所有份额后立即移除
  - 移除时机 2：项目结束时立即移除所有用户标签
- **平台购车用户标签**：
  - 添加条件：平台购车参与众筹项目购买比例获取
  - 添加时机：平台购车参与众筹时自动添加
  - 业务意义：平台作为特殊用户参与分润，享受相应权益
- 其他标签：运营人员手动添加或系统自动识别
- 标签可以叠加，一个用户可拥有多个标签

#### 4.1.5 常用人管理

**功能描述**：查看和管理用户添加的常用人信息。

**查看内容**：

- 常用人姓名、电话、身份证号
- 驾照信息（如已添加）
- 关系标签（本人/家人/朋友）
- 添加时间、使用次数

**管理操作**：

- 查看用户的所有常用人
- 查看常用人使用记录
- 协助用户处理常用人问题

**业务规则**：

- 每个用户最多 10 个常用人
- 常用人资料由用户自行管理
- 管理端只提供查看功能

#### 4.1.6 用户信用评估与风控管理

**功能描述**：建立用户信用评估体系，识别和管理用户风险，保障平台和车辆安全。

**信用评分体系**：

- **评分维度**：
  - 订单履约率：按时取还车、无违约记录
  - 支付信用：无逾期支付、无退款纠纷
  - 车辆使用：无事故、无违章、无损坏
  - 评价反馈：用户评价、平台评价
  - 资料完整性：实名认证、驾照认证完整性
- **评分规则配置**：
  - 各维度权重配置
  - 加分项规则（如：长期租车、按时还款）
  - 扣分项规则（如：违约、事故、投诉）
  - 评分计算公式
- **信用等级划分**：
  - 优秀（≥90 分）：享受优先服务、免押金等权益
  - 良好（80-89 分）：正常服务
  - 一般（60-79 分）：正常服务，需关注
  - 较差（40-59 分）：限制服务，需押金或担保
  - 差（<40 分）：列入黑名单，禁止使用服务

**风险评估规则配置**：

- **风险因子识别**：
  - 新用户风险：注册时间短、无历史记录
  - 违约风险：历史违约记录、逾期支付
  - 事故风险：历史事故记录、投诉记录
  - 欺诈风险：异常行为、资料异常
- **风险等级划分**：
  - 低风险：正常用户，无风险提示
  - 中风险：存在潜在风险，需关注
  - 高风险：存在明显风险，需限制
  - 极高风险：列入黑名单，禁止服务
- **风险评估规则**：
  - 自动评估规则配置
  - 手动评估流程
  - 风险等级更新机制

**黑名单管理**：

- **黑名单类型**：
  - 信用黑名单：信用评分过低
  - 欺诈黑名单：存在欺诈行为
  - 事故黑名单：重大事故责任方
  - 恶意用户：恶意投诉、恶意差评
- **黑名单管理**：
  - 加入黑名单：手动加入或自动加入
  - 黑名单原因记录
  - 黑名单有效期管理
  - 黑名单申诉处理
  - 移除黑名单流程
- **黑名单效果**：
  - 禁止预订车辆
  - 禁止参与活动
  - 禁止提现（如有余额）
  - 系统自动拦截

**异常行为监控**：

- **监控指标**：
  - 频繁取消订单（短时间内多次取消）
  - 异常支付行为（支付失败率高）
  - 异常投诉（恶意投诉）
  - 资料异常（身份证异常、驾照异常）
- **监控规则配置**：
  - 异常行为阈值设置
  - 监控频率配置
  - 告警规则配置
- **异常处理流程**：
  - 自动标记异常用户
  - 人工审核异常行为
  - 采取限制措施
  - 记录处理结果

**风险控制措施**：

- **预防措施**：
  - 新用户限制：首次租车需提供担保或押金
  - 信用门槛：低信用用户需额外押金
  - 实名认证：强制实名认证
  - 驾照验证：强制驾照验证
- **限制措施**：
  - 订单限制：高风险用户限制预订
  - 支付限制：需预付费或提高押金
  - 服务限制：限制某些服务使用
- **解除限制**：
  - 信用提升后自动解除
  - 人工审核解除
  - 申诉成功后解除

**信用修复机制**：

- **修复途径**：
  - 按时履约：正常完成订单可提升信用
  - 主动改善：主动解决历史问题
  - 申诉成功：申诉成功可恢复信用
  - 时间衰减：随着时间推移，历史负面记录影响减弱
- **修复规则配置**：
  - 修复加分规则
  - 修复周期设置
  - 修复上限设置

**业务规则**：

- 信用评分实时更新，订单完成后自动计算
- 高风险用户自动标记，运营人员人工审核
- 黑名单用户系统自动拦截，禁止服务
- 信用修复需审核，确保真实有效
- 所有信用评估和风险控制操作记录日志

**统计报表**：

- 用户信用分布统计
- 风险用户统计
- 黑名单统计
- 信用修复统计
- 风险控制效果分析

### 4.2 车辆管理功能

#### 4.2.1 功能概述

**管理范围**：

- 车型模板管理（标准化车型信息）
- 具体车辆管理（实际车辆档案）
- 车辆状态实时管理
- 车辆定期检查管理
- 车辆保养计划管理
- 车辆年检与证件管理
- 车辆库存与可用性管理
- 车辆评价审核和回复

#### 4.2.2 车型模板管理

**业务场景**：

- **场景 1**：内容管理员需要创建新车型模板，统一管理车型信息
- **场景 2**：门店新增车辆时，需要选择已存在的车型模板作为基础
- **场景 3**：运营人员需要更新车型信息（价格、配置等），影响所有同车型车辆
- **场景 4**：内容管理员需要维护车型图片和详情，提升展示效果

**功能描述**：创建和管理标准化的车型模板，为具体车辆提供基础信息。

**用户使用流程**：

1. 内容管理员登录管理端，进入车辆管理 → 车型模板管理
2. 创建新车型模板：
   - 填写基本信息（车型名称、品牌、型号、分类）
   - 配置车辆参数（座位数、床位数等）
   - 勾选设施配置清单
   - 上传车型图片（外观、内饰、设施、全景）
   - 编辑车型详情（富文本编辑，介绍、使用说明、特色亮点）
   - 设置默认价格（日租价、周租价、月租价、押金）
3. 保存车型模板，模板生效后可供门店选择
4. 门店添加车辆时，选择车型模板，系统自动填充基础信息
5. 如需修改车型信息，在模板中修改后，影响所有关联该模板的车辆

**车型模板内容**：

- **基本信息**：车型名称、品牌、型号、分类（B 型/C 型/拖挂式）
- **车辆参数**：座位数、床位数、车身尺寸、整备质量
- **设施配置**：厨房、卫浴、空调、冰箱、电视等设施清单
- **车型图片**：外观图、内饰图、设施图、全景图
- **车型详情**：富文本编辑的详细介绍、使用说明、特色亮点
- **默认价格**：基础日租价、周租价、月租价、押金

**功能价值**：

- **统一管理**：车型信息集中管理，保证信息一致性
- **提升效率**：门店添加车辆时复用模板，减少重复录入
- **灵活配置**：门店可在模板基础上调整实际配置，满足差异化需求
- **展示优化**：车型详情和图片统一维护，提升客户端展示效果

**业务规则**：

- 车型模板作为标准配置，具体车辆引用模板
- 门店添加车辆时可在模板基础上勾选实际配置
- 车型详情支持富文本编辑，显示在客户端详情页
- 车型图片和详情由内容管理员维护
- 修改车型模板影响所有关联该模板的车辆

**客户端影响**：

- 车型详情页的"车辆图文详情"区域显示车型详情
- 车辆配置列表显示车型配置信息

#### 4.2.3 具体房车管理

**业务场景**：

- **场景 1**：门店需要添加新车，选择车型模板并填写车辆信息
- **场景 2**：车辆需要维修，运营人员需要将车辆状态改为"维护中"
- **场景 3**：车辆完成维护，运营人员需要恢复车辆为"可用"状态
- **场景 4**：车辆需要调度到其他门店，运营人员需要更新车辆所属门店
- **场景 5**：车辆发生事故，运营人员需要记录事故信息和处理结果

**功能描述**：管理实际房车的档案信息、状态、维护记录。

**用户使用流程**：

1. 门店经理或运营管理员登录管理端，进入车辆管理 → 具体车辆管理
2. 添加新车：
   - 选择车型模板（系统自动填充基础信息）
   - 填写车辆基本信息（车牌号、VIN 码）
   - 设置所有权类型（合作车辆/众筹车辆）
   - 关联门店（必填，1 对 1 关系）
   - 勾选实际配置（在模板基础上调整）
   - 确认车辆状态为"可用"
3. 管理车辆状态：
   - 车辆需要维修：将状态改为"维护中"，车辆不再接受预订
   - 车辆维修完成：将状态改为"可用"，恢复接受预订
   - 车辆停用：将状态改为"停用"，车辆不再显示
4. 记录维护信息：
   - 记录维护时间、内容、费用
   - 更新里程数、油量、车况评估
   - 记录维护人员和门店
5. 车辆调度：
   - 如需调度车辆，更新车辆所属门店
   - 记录调度时间、原因、费用
   - 原始归属门店保持不变（用于收益分配）

**功能价值**：

- **车辆档案完整**：全面记录车辆信息，支持追溯和查询
- **状态实时管理**：车辆状态实时更新，确保可用性准确
- **维护记录追踪**：完整记录维护历史，支持车辆价值评估
- **调度管理便捷**：支持车辆调度，优化车辆资源配置

**房车档案信息**：

- **基本信息**：车牌号、VIN 码、关联车型模板（必填）、所属门店（必填，1 对 1 关系）
- **所有权信息**：
  - **所有权类型**：合作车辆（cooperative）/ 众筹车辆（crowdfunding）
  - **合作车辆**（cooperative）：
    - 来源：同业车辆（来自其他租赁平台或车行）
    - 取还车服务：由合作方门店完成
    - 门店类型：必须关联"合作商户"门店
    - 收益分配：
      - **收益方式**：平台通过第三方订单平台完成差价获利（非平台服务费）
      - **订单处理**：本项目只记录订单发生及指引完成订单取还车流程
      - **分润计算**：无需计算任何分润
      - **差价记录**：在订单管理中记录第三方平台实际订车费用，系统自动计算获利差价（获利差价 = 平台展示价格 - 第三方平台实际订车费用）
      - **订单服务指引**：
        - 同一城市的直营店或加盟店优先负责指引
        - 如果合作门店所在城市没有直营店或加盟店，则由总部客服对接或总部分配客服对接
        - 服务等级：同一城市 > 总部逐级提升
  - **众筹车辆**（crowdfunding）：
    - 来源：众筹项目（通过众筹资金采购）
    - 取还车服务：由平台直营/加盟门店完成
    - 门店类型：必须关联直营门店或加盟门店
    - 收益分配：参与众筹分润
    - 特殊情况：平台购车也参与众筹购买比例获取，标记为"平台购车用户"
- **房车状态**：可用/已租/维护中/停用
- **实际配置**：基于车型模板勾选的实际配置
- **价格来源**：使用车型默认价格或价格策略管理
- **门店关联**：
  - storeId：车辆所属门店（必填，1 对 1 关系）
  - originalStoreId：原始归属门店（用于收益分配，调度后保持不变）
  - isRemoteTask：异地任务标识（临时标记，用于异地还车任务）
  - partnerStoreId：合作车辆专用，合作方门店 ID

**房车维护记录**：

- 维护时间、维护内容、维护费用
- 里程数、油量、车况评估
- 维护人员、维护门店

**房车调度记录**：

- 调度时间、调出门店、调入门店
- 调度原因、费用分摊
- 原始归属门店（保持不变）

**车辆调度与收益分配规则**：

**场景 1：总部划拨（永久调度）**：

- storeId 更新为新门店 ID
- originalStoreId 保持为原门店 ID（不变）
- 门店分润：按划拨后的新门店分配 10% 分润
- 收益分配：按划拨日期划分，精确到日

**场景 2：异地还车任务（临时任务）**：

- 设置 isRemoteTask = true（临时标记）
- storeId 不变（仍为原门店）
- **门店分润规则**（修正）：

  - 车辆在哪个门店执行订单，10% 门店分润就计入该门店

  - 统一规则：按订单执行门店分配 10% 分润

- **异地还车服务费归属**：
  - 不参与分润，作为费用归属直接分配给门店
  - 分配规则：50% 异地门店，50% 原门店
  - 归属方式：费用归属，不计入分润体系
- **众筹车主分润**：
  - 不受车辆调度影响
  - 仍按订单收入计算，不受门店收益分配影响

**业务规则**：

- 房车必须关联车型模板（vehicleModelId 必填）
- 房车必须关联门店（storeId 必填，1 对 1 关系）
- **车辆入库时，如车型模板不存在，必须先创建车型模板再入库**
- 合作车辆必须关联合作商户门店
- 众筹车辆必须关联直营门店或加盟门店
- **众筹车辆（ownershipType='crowdfunding'）必须关联一个众筹项目（1 对 1 关系）**
- **合作车辆订单的差价计算在订单管理中完成，不参与分润体系**
- 房车价格统一使用车型默认价格或价格策略
- 房车调度后原始归属不变，用于收益分配
- 维护中的房车不显示在小程序端

#### 4.2.4 房车评价管理

**功能描述**：审核用户提交的房车评价，回复用户评价，管理评价展示。

**评价审核**：

- 查看待审核评价列表

- 评分计算：所有评价的平均分
- 评价排序：最新评价优先

**业务规则**：

- 评价必须审核通过才显示
- 评价一经发布不可修改
- 恶意评价可删除并说明原因
- 评价聚合让用户更全面了解车型

**小程序端对应**：

- 房车详情页的"用户评价"区域显示该车型的所有评价

- 标签图标

#### 4.2.6 车辆定期检查管理

**功能描述**：管理车辆定期安全检查，确保车辆安全合规运营。

**检查计划配置**：

- **检查类型**：
  - 每日检查：外观、基本功能、安全设施
  - 周检查：深度清洁、功能测试、设备检查
  - 月检查：全面检查、性能测试、隐患排查
  - 季度检查：全面检修、系统测试、安全评估
- **检查规则配置**：
  - 按里程数：每行驶 X 公里需检查
  - 按时间：每 X 天需检查
  - 按订单数：每完成 X 单需检查
  - 混合规则：同时满足里程和时间要求
- **检查项目模板**：
  - 外观检查：车身、玻璃、轮胎、灯光
  - 功能检查：发动机、刹车、转向、空调、电器
  - 安全设施：安全带、灭火器、警示牌
  - 随车物品：工具包、备胎、说明书

**检查任务生成**：

- 系统自动生成检查任务
- 提前 3 天发送检查提醒
- 任务分配给门店或指定人员
- 支持手动创建检查任务

**检查任务管理**：

- **任务列表**：
  - 待检查任务列表
  - 进行中任务列表
  - 已完成任务列表
  - 逾期任务列表
- **任务筛选**：
  - 按检查类型筛选
  - 按车辆筛选
  - 按门店筛选
  - 按任务状态筛选
- **任务详情**：
  - 车辆信息
  - 检查项目清单
  - 检查标准说明
  - 完成时间要求

**检查结果记录**：

- **检查项目逐项记录**：
  - 每个检查项目的结果（正常/异常）
  - 异常项目详细描述
  - 拍照记录异常情况
  - 检查人员签字
- **检查结论**：
  - 检查通过：车辆可正常使用
  - 检查不通过：需维修或保养
  - 需整改：存在隐患需及时处理
- **整改跟踪**：
  - 不通过项整改计划
  - 整改责任人
  - 整改完成时间
  - 整改验收确认

**业务规则**：

- 检查任务必须按时完成，逾期自动提醒
- 检查不通过的车辆自动标记为"维护中"，不可出租
- 检查结果记录永久保存，作为安全追溯依据
- 检查人员必须具有相应资质
- 严重安全隐患必须立即处理，不能出租

**预警机制**：

- 检查任务即将到期提醒（提前 3 天）
- 检查任务逾期提醒（当天）
- 检查不通过车辆预警
- 整改超时预警

#### 4.2.7 车辆保养计划管理

**功能描述**：管理车辆定期保养计划，确保车辆良好状态，延长使用寿命。

**保养计划模板配置**：

- **保养类型**：
  - 日常保养：清洁、检查、调整
  - 定期保养：机油、机滤、空滤更换
  - 大保养：全面检修、部件更换
  - 季节性保养：根据季节特点的特殊保养
- **保养项目配置**：
  - 保养项目名称
  - 保养周期（里程/时间）
  - 保养费用预估
  - 保养所需时间
  - 是否必需项目
- **保养规则配置**：
  - 首保规则：新车 X 公里或 X 个月首次保养
  - 定期保养：每 X 公里或每 X 个月保养一次
  - 特殊保养：特殊条件下的保养要求

**保养计划生成**：

- 系统根据车辆里程和时间自动生成保养计划
- 支持按车型批量生成保养计划
- 支持手动创建保养计划
- 保养计划包含：保养类型、预计时间、预计费用、执行门店

**保养任务管理**：

- **任务列表**：
  - 待执行保养任务
  - 进行中保养任务
  - 已完成保养任务
  - 逾期保养任务
- **任务详情**：
  - 车辆信息（车牌号、车型、当前里程）
  - 保养项目清单
  - 保养标准要求
  - 预估费用和时间
  - 执行门店和责任人
- **任务优先级**：
  - 高优先级：影响安全的保养项目
  - 中优先级：影响性能的保养项目
  - 低优先级：常规保养项目

**保养执行记录**：

- **保养过程记录**：
  - 保养开始时间
  - 实际保养项目（可能有调整）
  - 保养中发现的问题
  - 保养照片记录
  - 保养人员信息
- **保养结果记录**：
  - 保养完成时间
  - 实际保养费用
  - 更换的零部件清单
  - 保养后车辆状态评估
  - 下次保养提醒时间
- **保养费用管理**：
  - 费用明细（人工费、材料费、其他费用）
  - 费用归属（平台/门店/供应商）
  - 费用审批流程

**业务规则**：

- 保养任务必须按时执行，逾期自动提醒和升级
- 保养中的车辆自动标记为"维护中"，不可出租
- 保养完成后必须验收，验收通过才能恢复可用状态
- 保养记录永久保存，用于车辆价值评估和保险理赔
- 保养费用必须记录完整，用于成本核算

**预警机制**：

- 保养任务即将到期提醒（提前 7 天）
- 保养任务逾期提醒（当天）
- 保养超时预警（执行时间超过预期）
- 保养费用超预算预警

**统计报表**：

- 车辆保养完成率统计
- 保养费用统计分析
- 保养周期分析
- 保养效果评估

  - 保险证件照片上传
  - 保险到期提醒（提前 30 天）
- **其他证件管理**：
  - 道路运输证（如适用）
  - 环保标志（如适用）
  - 进京证等临时证件记录

**证件到期提醒**：

- 证件到期前 30 天提醒
- 证件到期前 7 天提醒
- 证件已过期警告
- 证件到期自动标记车辆状态（如：证件过期车辆不可出租）

**业务规则**：

- 所有证件信息必须完整录入
- 证件照片必须清晰可辨认
- 证件到期车辆自动标记为"停用"，不可出租
- 证件更新后必须及时上传新证件
- 证件信息变更必须记录变更历史

**预警机制**：

- 证件即将到期提醒
- 证件已过期警告
- 年检即将到期提醒
- 年检已过期警告

**统计报表**：

- 证件到期统计（即将到期/已过期）
- 年检完成率统计
- 证件合规率统计

#### 4.2.9 车辆库存与可用性管理

**功能描述**：精细化管理车辆库存，实时掌握车辆可用状态，优化车辆分配和调度。

**库存统计管理**：

- **按门店统计**：
  - 门店总车辆数
  - 可用车辆数（可出租）
  - 已租车辆数（租赁中）
  - 维护中车辆数
  - 停用车辆数
  - 车辆利用率 = 已租车辆数 ÷ 总车辆数
- **按车型统计**：
  - 各车型总数量
  - 各车型可用数量
  - 各车型预订占用数量
  - 各车型库存预警
- **实时库存更新**：
  - 订单确认后自动占用库存
  - 订单取消后自动释放库存
  - 车辆状态变更时自动更新库存
  - 车辆调度时更新相关门店库存

**可用性管理**：

- **可用车辆定义**：
  - 车辆状态为"可用"
  - 无待执行的检查或保养任务
  - 证件齐全且在有效期内
  - 无订单占用
- **预订占用管理**：
  - 预订确认后占用车辆库存
  - 预订取消后释放库存
  - 预订占用时间范围管理
  - 防止超售机制
- **可用时间窗口管理**：
  - 每个车辆的可用时间段
  - 考虑维护、检查、调度的占用时间
  - 实时计算车辆最早可用时间

**库存预警机制**：

- **库存不足预警**：
  - 按车型：某车型可用数量 ≤ 设定阈值（如 2 辆）
  - 按门店：某门店总可用车辆数 ≤ 设定阈值
- **库存异常预警**：
  - 长期无订单的车辆（可能位置不佳或价格不合理）
  - 频繁维护的车辆（可能车况不佳）
  - 库存积压预警（某门店库存过多）
- **预警通知**：
  - 站内消息通知运营人员
  - 邮件通知门店经理
  - 紧急情况电话通知

**车辆分配策略**：

- **自动分配规则**：
  - 优先分配可用车辆最多的门店
  - 优先分配距离用户最近的车辆
  - 考虑车辆使用率均衡
  - 考虑车辆维护计划
- **手动分配**：
  - 运营人员可手动指定车辆
  - 支持批量分配
  - 记录分配原因

**库存报表**：

- **实时库存报表**：
  - 各门店库存统计
  - 各车型库存统计
  - 库存利用率分析
- **库存趋势分析**：
  - 库存变化趋势图
  - 库存周转率分析
  - 季节性库存变化
- **库存优化建议**：
  - 库存不均衡提醒
  - 库存配置建议
  - 车辆调度建议

**业务规则**：

- 库存数据实时更新，保证准确性
- 预订占用必须与实际订单状态同步
- 库存不足时优先显示其他可用车辆或建议调整取车时间
- 库存数据用于车辆采购和调配决策
- 库存报表支持导出和分享

#### 4.3.3 订单退款处理

**业务场景**：

- **场景 1**：用户申请退款，财务人员需要审核退款原因和金额
- **场景 2**：用户取消订单，系统自动创建退款申请，财务人员需要处理
- **场景 3**：订单异常导致退款，运营人员需要特殊处理退款流程

**功能描述**：处理用户的退款申请，审核退款金额，执行退款操作。

**用户使用流程**：

1. 用户申请退款或订单取消，系统自动创建退款申请
2. 财务人员进入订单管理 → 退款处理，查看待处理退款列表
3. 审核退款申请：
   - 查看订单详情和退款原因
   - 检查退款金额计算是否正确
   - 验证退款是否符合退款规则
4. 审批退款：
   - 审批通过：确认退款金额，执行退款操作，退款原路退回
   - 审批拒绝：填写拒绝原因，通知用户
5. 退款完成后，订单状态更新，相关费用和库存自动处理

**功能价值**：

- **退款合规**：确保退款符合业务规则，保障用户权益
- **快速处理**：退款流程清晰，处理效率高
- **风险控制**：审核机制确保退款安全

**退款申请审核流程**：

1. 查看退款申请列表
2. 查看退款详情（订单信息、退款原因、退款金额）
3. 核实订单状态和取消时间
4. 根据取消政策计算应退金额
5. 审核通过或拒绝（拒绝需说明原因）
6. 审核通过后系统自动退款

**退款金额计算**：

- 取车前 24 小时以上：全额退款
- 取车前 24 小时内：扣除 10%手续费
- 取车后退款：根据实际使用天数计算

**业务规则**：

- 退款申请必须财务管理员审核
- 退款金额根据取消政策自动计算，可手动调整
- 退款原路返回原支付方式
- 退款记录完整保留备查

#### 4.3.4 续租申请审核

**业务场景**：

- **场景 1**：用户希望延长租期，提交续租申请
- **场景 2**：运营人员需要审核续租申请，检查车辆可用性
- **场景 3**：续租审核通过后，需要用户支付续租费用

**功能描述**：审核用户的续租申请，确认车辆可用性，调整订单和费用。

**用户使用流程**：

1. 用户提交续租申请（在原订单还车日期前）
2. 运营人员进入订单管理 → 续租审核，查看待审核续租申请
3. 审核续租申请：
   - 查看续租申请详情（原订单、续租天数、续租原因）
   - 检查车辆续租期间是否有预订冲突
   - 计算续租费用（按当前价格策略）
   - 验证用户是否符合续租条件
4. 审批续租：
   - 审批通过：通知用户审核结果和续租费用，等待用户支付
   - 审批拒绝：填写拒绝原因，通知用户
5. 用户支付续租费用后，订单还车时间自动更新

**功能价值**：

- **灵活服务**：满足用户临时延长租期的需求，提升用户体验
- **资源管控**：通过审核机制确保车辆资源合理分配
- **收入增长**：续租带来额外收入，提升业务收益

**续租审核流程**：

1. 查看续租申请列表
2. 查看申请详情（原订单、续租天数、续租原因）
3. 检查车辆续租期间是否有预订冲突
4. 计算续租费用
5. 审核通过或拒绝
6. 通知用户审核结果和费用
7. 用户支付续租费用
8. 更新订单还车时间

**续租条件**：

- 必须在原订单还车日期前提交申请
- 续租期间车辆无其他预订冲突
- 用户无订单违约或事故欠费记录
- 续租价格按当前价格策略计算

**业务规则**：

- 续租申请必须提前 24 小时
- 续租最长不超过原订单租期
- 续租价格可能与原订单价格不同（根据当前策略）
- 续租费用单独支付，不合并原订单

#### 4.3.5 事故报告处理

**功能描述**：处理用户报告的车辆事故，协调保险理赔，记录事故处理过程。

**事故处理流程**：

1. 接收用户事故报告
2. 查看事故详情（时间、地点、现场照片、事故描述）
3. 联系用户了解详情
4. 判断事故严重程度（轻微/中度/严重）
5. 指导用户处理（报警/保险定损/现场等待）
6. 协调保险公司理赔
7. 评估责任和费用
8. 记录事故处理过程
9. 更新订单状态和费用

**事故分类**：

- **轻微事故**：轻微剐蹭，不影响使用，维修费<1000 元
- **中度事故**：部分损坏，需维修，维修费 1000-5000 元
- **严重事故**：严重损坏或人员伤亡，需报警和保险处理

**责任判定**：

- 用户责任：用户操作不当导致的事故
- 平台责任：车辆本身质量问题
- 不可抗力：自然灾害、第三方责任等

**业务规则**：

- 事故必须在 24 小时内报告
- 严重事故必须报警和保险理赔
- 责任判定影响用户赔偿金额
- 保险范围内的损失由保险赔付

#### 4.3.6 合作车辆订单管理（新增）

**功能描述**：管理合作车辆的订单，记录第三方平台实际订车费用，计算平台获利差价，指引订单取还车流程。

**订单记录**：

- **订单基本信息**：
  - 订单号、订单类型（合作车辆订单）
  - 用户信息（昵称、手机号）
  - 车辆信息（车牌号、车型）
  - 租期、订单金额
  - 取还车门店（合作商户门店）
  - 创建时间、订单状态
- **第三方平台信息**：
  - 第三方平台名称（如：携程、飞猪、途牛等）
  - 第三方平台订单号
  - **第三方平台实际订车费用**（必填）：记录从第三方平台获得的实际订单金额
  - 平台展示价格：在本项目中小程序端展示的价格
- **差价计算**：
  - 平台展示价格：用户支付的订单金额
  - 第三方平台实际订车费用：支付给合作方的实际费用
  - **获利差价 = 平台展示价格 - 第三方平台实际订车费用**
  - 差价记录：自动计算并记录，用于财务统计
- **订单服务指引**：
  - **指引门店分配规则**：
    - 优先级 1：同一城市的直营店或加盟店负责指引
    - 优先级 2：如果合作门店所在城市没有直营店或加盟店，则由总部客服对接或总部分配客服对接
    - 服务等级：同一城市 > 总部逐级提升
  - **指引内容**：
    - 订单确认和提醒
    - 取车流程指引（时间、地点、注意事项）
    - 还车流程指引（时间、地点、注意事项）
    - 订单问题处理
  - **指引记录**：
    - 指引门店/客服信息
    - 指引时间、指引内容
    - 用户反馈和问题处理记录

**业务流程**：

1. **订单创建**：

   - 用户在小程序端下单（合作车辆）
   - 记录订单基本信息
   - 获取第三方平台订单信息
   - 录入第三方平台实际订车费用
   - 系统自动计算获利差价

2. **订单服务指引分配**：

   - 系统自动识别合作门店所在城市
   - 查找同一城市的直营店或加盟店
   - 如果有：分配给该门店负责指引
   - 如果没有：分配给总部客服或总部分配客服对接

3. **订单执行**：

   - 指引门店/客服发送取车指引
   - 用户在合作门店完成取车
   - 订单状态更新为"使用中"
   - 指引门店/客服发送还车指引
   - 用户在合作门店完成还车
   - 订单状态更新为"已完成"

4. **订单完成**：
   - 记录订单完成信息
   - 计算并记录获利差价
   - 生成财务统计记录

**业务规则**：

- 合作车辆订单必须记录第三方平台实际订车费用
- 获利差价自动计算，不允许手动修改
- 订单服务指引优先分配给同一城市的直营店或加盟店
- 如果城市内没有直营店或加盟店，由总部客服对接
- 订单状态流转：待支付 → 待取车 → 使用中 → 待还车 → 已完成
- **合作车辆订单不参与任何分润计算**
- 差价记录用于财务统计，不影响分润体系

**财务统计**：

- 按订单统计获利差价
- 按时间段统计合作车辆订单总差价
- 按门店统计合作车辆订单数量
- 导出财务报表

**小程序端对应**：

- 用户在订单详情页查看订单信息
- 用户接收取还车指引通知
- 用户查看订单状态和流程

#### 4.4.6 收益分润管理

**功能描述**：管理众筹项目的收益分配，基于单笔订单完成时计算分润，次月 5 日统一划拨。

**分润计算方式**（单笔订单）：

```
单笔订单可分润金额 = 订单金额 - 不参与分润的费用

单笔订单运营成本 = 订单对应的（保险费 + 维护费 + 清洁费 + 平台服务费）

单笔订单净收益 = 单笔订单可分润金额 - 单笔订单运营成本

众筹车主单笔分润 = 单笔订单净收益 × 30% × 个人份额比例
```

**分润计算时机**（修正）：

- **订单完成时立即计算**：
  - 订单状态变更为"已完成"时，立即计算该订单的众筹车主分润
  - 分润记录保存到众筹车主账户，状态为"待划拨"
  - 每个分润记录必须关联订单号
- **次月 5 日统一划拨**：
  - 每月 5 日统一将上月所有订单的分润划拨到众筹车主钱包
  - 生成订单清单，清晰展示每笔分润来源
  - 更新分润记录状态为"已划拨"

**月保底订单收入配置**（可选）：

- **配置方式**：后台可设置车辆的月保底订单收入（如：1000 元/月/车）
- **配置属性**：自定义、非必要项（可选配置）
- **配置位置**：在车辆上线时或运营启动前配置
- **处理方式**（月度汇总处理）：
  1. 每月 1 日统计上月所有订单收入总和
  2. 如果总订单收入 < 月保底订单收入：
     - 计算保底差额 = 月保底订单收入 - 总订单收入
     - 生成保底差额订单记录（虚拟订单）
     - 按保底差额计算额外分润，在次月 5 日一并划拨
  3. 如果总订单收入 ≥ 月保底订单收入：按实际订单收入计算，不产生保底差额
- **修改规则**：运营中可修改，修改后下月生效
- **保底差额成本承担**：需明确配置（建议由平台承担差额成本）

**计算示例**：

```
场景1：设置了保底收入（1000元/月）
  - 当月实际收入：1500元
  - 月运营成本：200元
  - 月净收益 = max(1500, 1000) - 200 = 1300元
  - 个人分润 = 1300 × 份额比例

场景2：设置了保底收入（1000元/月）
  - 当月实际收入：800元
  - 月运营成本：200元
  - 月净收益 = max(800, 1000) - 200 = 800元（使用保底收入）
  - 个人分润 = 800 × 份额比例

场景3：未设置保底收入
  - 当月实际收入：800元
  - 月运营成本：200元
  - 月净收益 = 800 - 200 = 600元（按实际收入）
  - 个人分润 = 600 × 份额比例
```

**分润流程**（修正）：

1. **订单完成时**：

   - 订单状态变更为"已完成"时，立即计算该订单的众筹车主分润
   - 生成分润明细记录（关联订单号，状态：待划拨）
   - 保存到众筹车主账户

2. **次月 1 日（保底收入处理）**：

   - 统计上月所有订单收入总和
   - 如果设置了保底收入且总订单收入 < 保底收入：
     - 计算保底差额并生成虚拟订单记录
     - 计算保底差额分润

3. **次月 5 日统一划拨**：
   - 系统自动汇总上月所有订单的分润记录（含保底差额）
   - 生成订单清单（每个众筹车主一份）
   - 批量划拨到众筹车主钱包
   - 更新分润记录状态为"已划拨"
   - 发送分润通知和订单清单

**分润规则**：

- **分润基于单笔订单完成时计算**，不按月汇总计算
- **次月 5 日统一划拨**上月所有订单的分润
- 每个分润记录必须关联订单号
- 用户可查看每笔分润的订单来源
- 亏损订单不分润（订单净收益 ≤ 0 时）
- **分润起始时间：从项目启动运营的当月开始计算（按实际天数）**
- **计算对象：只计算已还车完成的订单（按订单完成时间统计）**

**订单清单格式**：

- 每个众筹车主收到订单清单，包含：
  - 订单号（必填，关联具体订单）
  - 订单完成日期
  - 订单金额
  - 分润金额
  - 分润来源说明（如：众筹车主分润）
- 清单支持导出和查询

**小程序端对应**：

- 用户在"我的份额"查看持有车辆和收益
- 查看每月分润明细和历史记录

#### 4.4.8 车辆采购管理

**功能描述**：管理众筹项目成功后的车辆采购流程。

**采购流程**：

1. **采购申请**：

   - 项目状态为"成功"（达到最低成功份额）
   - 项目状态更新为"采购中"
   - 确认采购资金可用（完全使用众筹资金）
   - 一个项目采购一台车辆（1 对 1 关系）

2. **采购执行**：

   - 使用众筹资金采购车辆
   - 采购资金完全来自众筹资金，无需平台垫付
   - 记录采购合同、发票等凭证
   - 跟踪车辆采购进度

3. **采购完成**：
   - 车辆到货后，进入车辆入库流程
   - 采购记录完整保存备查

**业务规则**：

- 一个众筹项目对应一台车辆（1 对 1 关系）
- 采购资金完全使用众筹资金，无需平台垫付
- 采购合同、发票等凭证必须完整保存

#### 4.4.9 车辆入库与项目关联管理

**功能描述**：车辆到货后，创建车辆档案并关联众筹项目。

**入库流程**：

1. **到货验收**：

   - 检查车辆外观、内饰、功能
   - 核对采购单信息
   - 拍照记录
   - 确认无误后验收

2. **创建车辆档案**：

   - **关联车型模板**（必填）：
     - 如车型模板已存在：直接关联
     - 如车型模板不存在：必须先创建车型模板，再创建车辆档案
   - 录入车辆基本信息（车牌号、VIN 码等）
   - 设置车辆所有权类型：`ownershipType='crowdfunding'`
   - **注意：此时不指定门店，待上线时指定**
   - 设置车辆状态：`status='pending_operation'`（待上线）

3. **关联众筹项目**：
   - 选择对应的众筹项目（状态为"采购中"）
   - 建立关联：`project.vehicleId = vehicle.id`
   - 项目状态更新为：`status='pending_operation'`（待上线）
   - **注意：此时项目的 storeId 仍为空，待上线时指定**

**业务规则**：

- 车辆必须关联车型模板（vehicleModelId 必填）
- 如车型模板不存在，必须先创建车型模板再入库
- **车辆入库时不指定门店（待上线环节指定）**
- 一个车辆只能关联一个众筹项目（1 对 1 关系）
- 一个众筹项目只能关联一个车辆（1 对 1 关系）
- 车辆入库后，项目状态自动更新为"待上线"

#### 4.4.9.1 车辆上线与门店指定管理

**功能描述**：车辆入库后，指定运营门店，准备启动运营。

**上线操作流程**：

1. **选择运营门店**：

   - 从直营门店或加盟门店中选择（不能选择合作商户）
   - 查看门店信息和车辆容量
   - 确认门店可接受该车辆

2. **指定门店归属**：

   - 设置车辆所属门店：`vehicle.storeId = 选择的门店ID`
   - 设置原始归属门店：`vehicle.originalStoreId = 选择的门店ID`（初始值相同）
   - 设置项目运营门店：`project.storeId = 选择的门店ID`
   - 更新车辆状态：`status='available'`

3. **配置月保底订单收入**（可选）：

   - 可设置车辆的月保底订单收入（如：1000 元/月/车）
   - 配置属性：自定义、非必要项
   - 可在运营启动前或上线时配置

4. **确认上线**：
   - 二次确认门店选择
   - 记录上线时间和操作人
   - 项目状态更新为：`status='pending_operation'`（待运营）

**业务规则**：

- 车辆上线时必须指定门店（storeId 必填）
- 门店类型必须是直营门店或加盟门店
- 车辆上线后，项目运营门店与车辆门店一致
- 上线操作不可撤销（需超级管理员权限才能调整）
- 所有操作记录完整日志

**小程序端对应**：

- 车辆在小程序端显示为"可用"状态
- 车辆归属门店信息同步显示

#### 4.4.10 运营启动确认管理

**功能描述**：车辆上线后，确认启动运营，开始添加标签、发放积分、启动分润。

**启动确认流程**：

1. **确认车辆状态**：

   - 车辆已上线并指定门店
   - 车辆状态为"可用"
   - 门店可正常提供服务

2. **启动运营操作**：

   - 确认启动运营
   - 项目状态更新为：`status='operating'`（运营中）
   - 自动执行以下操作：
     - **添加标签**：为所有持有该项目份额的用户添加"众筹车主"标签
     - **发放积分**：按积分发放计划为众筹车主发放初始积分
     - **启动分润**：启动分润计算系统，从当月开始计算

3. **记录启动信息**：
   - 启动时间
   - 启动操作人
   - 启动时的份额持有情况

**业务规则**：

- 车辆上线后需要确认启动运营
- 启动运营后，标签、积分、分润系统开始工作
- 分润从启动运营的当月开始计算（按实际天数）
- 启动操作不可撤销

**小程序端对应**：

- 用户在"我的份额"中可看到车辆已运营
- 用户标签和积分自动更新

#### 4.4.11 份额转让与权益处理管理

**功能描述**：管理份额转让时的标签、积分、权益处理。

**份额转让处理规则**：

1. **标签处理**：

   - **原车主卖出所有份额**：
     - 如果项目运营中：立即移除"众筹车主"标签
     - 如果项目已结束：立即移除"众筹车主"标签
   - **新买家获得份额**：
     - 如果项目运营中：立即添加"众筹车主"标签
     - 如果项目未运营：不添加标签（待运营启动时添加）
   - **标签添加时机**：根据项目是否运营决定（项目状态='operating'时添加）

2. **积分处理**：

   - **积分平移**：新买家继承原份额对应的积分账户
   - **原车主积分清零**：原车主的所有积分及相应奖励、权益即时清零
   - **不再单独发放**：购买他人份额的用户不再单独获取积分

3. **权益处理**：
   - 新买家享受份额对应的所有权益（分润等）
   - 原车主的所有权益即时取消

**业务规则**：

- 份额转让时标签、积分、权益自动处理
- 标签添加以项目是否运营为准
- 积分平移给新买家，原车主清零
- 所有处理即时生效

### 4.5 份额市场管理功能

#### 4.5.1 功能概述

**业务目标**：管理众筹份额的二级交易，审核挂牌申请，监控交易过程，处理交易纠纷。

**管理范围**：

- 挂牌出售审核
- 交易过程监控
- 份额过户处理
- 交易手续费配置
- 交易纠纷处理

#### 4.5.2 挂牌出售审核

**功能描述**：审核众筹车主提交的份额挂牌出售申请。

**审核内容**：

- 核实卖家持有份额情况
- 检查份额是否可转让（无冻结、无纠纷）
- 核对挂牌价格合理性
- 确认同一车辆份额整体转让

**审核标准**：

- 份额真实存在且属于申请人
- 价格在合理范围内（参考价 ±50%）
- 同车份额整体转让，不拆分
- 无其他限制条件

**审核流程**：

1. 查看挂牌申请列表
2. 查看申请详情
3. 核实份额信息
4. 审核通过或拒绝
5. 审核通过后份额上架市场

**业务规则**：

- 挂牌需要审核，防止异常交易
- 审核时限：24 小时内完成
- 拒绝需说明原因
- 份额上架后随时可撤销

#### 4.5.3 交易监控管理

**功能描述**：实时监控份额交易过程，及时发现和处理异常。

**监控内容**：

- 交易订单列表
- 交易状态（待支付/已支付/过户中/已完成/已取消）
- 交易双方信息
- 交易金额和手续费
- 资金流转记录

**异常预警**：

- 异常高价或低价交易
- 频繁交易同一份额
- 交易双方关联账号
- 支付异常或超时

**业务规则**：

- 所有交易需要监控
- 异常交易及时人工介入
- 保护买卖双方权益

#### 4.5.4 份额过户处理

**功能描述**：处理份额所有权转移，更新份额持有记录。

**过户流程**：

1. 买家支付成功后自动触发过户
2. 系统扣除卖家份额
3. 系统增加买家份额
4. 更新份额持有记录
5. 计算并扣除手续费
6. 转账给卖家（扣除手续费后）
7. 发送过户通知给买卖双方

**过户记录**：

- 过户时间
- 交易双方信息
- 份额数量、交易价格
- 手续费金额
- 过户前后份额持有情况

**业务规则**：

- 过户自动完成，无需人工干预
- 手续费从卖家收款中扣除
- 过户完成后不可撤销
- 完整记录备查

#### 4.5.5 手续费配置

**功能描述**：配置份额交易的手续费率和收费规则。

**配置内容**：

- 手续费率：例如 3%
- 最低手续费：例如 10 元
- 最高手续费：例如 1000 元
- 优惠政策：VIP 用户折扣

**业务规则**：

- 手续费从卖家收款中扣除
- 手续费配置实时生效
- 修改需要记录和审批

#### 4.5.6 交易纠纷处理

**功能描述**：处理份额交易中的纠纷和投诉。

**纠纷类型**：

- 支付纠纷：已支付未到账
- 过户纠纷：过户延迟或失败
- 价格纠纷：价格显示错误
- 欺诈纠纷：虚假交易

**处理流程**：

1. 接收纠纷投诉
2. 调查核实情况
3. 联系双方了解详情
4. 查看交易记录和证据
5. 做出处理决定
6. 执行处理方案（退款/补偿/警告）
7. 记录处理过程
8. 通知双方处理结果

**业务规则**：

- 纠纷必须在 7 天内提出
- 平台保持中立，依据事实处理
- 恶意投诉将受到处罚
- 所有处理有完整记录

### 4.6 优惠券管理功能

#### 4.6.1 功能概述

**业务目标**：创建和管理优惠券，促进用户转化，提升营销效果。

**管理范围**：

- 优惠券发放管理
- 优惠券使用统计
- 优惠券规则配置

#### 4.6.2 优惠券创建管理

**功能描述**：创建不同类型的优惠券，设置使用规则。

**优惠券类型**：

- **现金券**：直接抵扣固定金额
- **折扣券**：按比例折扣
- **满减券**：满 XX 元减 XX 元
- **日租抵扣券**：抵扣 X 天租金（房车租赁专用）

**配置内容**：

- **基本信息**：券名称、券类型、券面额/折扣率
- **使用规则**：
  - 使用门槛：满 XX 元可用
  - 适用场景：房车租赁/营地预订/旅行定制/全场通用
  - 有效期：发放后 X 天有效
  - 叠加规则：是否可与其他券叠加（由管理端设置）
- **发放设置**：
  - 发放方式：手动发放/活动发放/购买获得
  - 发放数量限制
  - 售价（如可购买）
- **转赠规则**：
  - 是否支持转赠
  - 转赠次数限制（仅可转赠一次）

**业务规则**：

- 未发放的券可随时编辑
- 已发放的券只可修改名称，不可修改规则
- 优惠券规则由管理端统一配置
- 同类型优惠券叠加规则由管理端定义

**小程序端对应**：

- 优惠券商城显示可购买的优惠券
- 我的优惠券显示用户持有的券
- 下单时可选择可用优惠券

#### 4.6.3 优惠券发放管理

**功能描述**：向用户发放优惠券，控制发放数量。

**发放方式**：

- **手动发放**：选择指定用户，手动发放
- **批量发放**：导入用户列表，批量发放
- **活动发放**：设置活动规则，自动发放
- **购买发放**：用户在小程序购买后自动发放

**发放记录**：

- 发放时间、发放方式
- 发放数量、接收用户
- 有效期、使用状态

**业务规则**：

- 发放后不可撤回
- 发放记录完整保留
- 超过有效期自动失效

#### 4.6.4 优惠券使用统计

**功能描述**：统计优惠券的使用情况，评估营销效果。

**统计维度**：

- 发放量、使用量、核销率
- 使用场景分布
- 用户使用行为分析
- 营销 ROI 计算

**数据展示**：

- 优惠券使用趋势图
- 不同类型券的使用对比
- 核销明细列表

#### 4.9.7 分润基数计算

**标准分润基数计算**：

1. 订单总额 = 租金 + Σ(参与分润的增值费用)
2. 优惠后金额 = 订单总额 - 优惠券抵扣
3. 可分润金额 = 优惠后金额 - Σ(不参与分润的费用)
4. 各方分润 = 可分润金额 × 配置的分润比例

**特殊推广用户分润计算**：

- **特殊推广用户的分润从订单总额内分润**（不是从可分润金额）
- 计算基数：订单可分润金额
- 分润比例：自定义（最高 10%）
- 说明：特殊推广用户的分润作为订单总额的一部分参与分润计算

#### 4.9.8 业务规则

- 费用归属在订单完成后触发
- 费用调整后对新订单立即生效
- 勾选"参与分润"的费用计入分润基数
- 保证金固定不参与分润
- 订单退款时所有费用原路退回
- 结算费用需门店还车时确认
- 异常费用需管理员审核
- 异地还车费用申请需审核后发放

**供应商类型**：

#### 4.10.4 供应商结算与对账

**功能描述**：管理供应商结算，定期对账，确保账务准确。

**结算管理**：

- **结算明细**：服务项目、服务时间、服务费用、其他费用、费用合计、税额、实付金额
- **结算审批**：结算单提交审批、财务审核、审批通过后付款

**对账管理**：

- 对账周期：每月对账、每季度对账

- 对账确认：供应商确认对账结果、对账差异处理流程

**付款管理**：

- 付款申请、付款审批、付款记录、发票管理

#### 4.10.5 供应商评价管理

**功能描述**：对供应商服务质量进行评价，优化供应商管理。

**评价维度**：

- 价格合理性：价格是否合理
- 响应速度：问题响应和处理速度
- 合作配合度：合作过程中的配合度

**评价管理**：

- 定期评价（每季度/每半年）、服务后即时评价、评价记录和统计、评价结果用于供应商分级

**供应商分级**：

- A 级供应商：优先合作，扩大合作范围
- B 级供应商：正常合作
- C 级供应商：关注改善，限制合作
- D 级供应商：考虑更换，终止合作

**业务规则**：

- 评价结果影响供应商合作等级

- 评价记录永久保存

#### 5.3.3 门店信息管理

**功能描述**：管理门店的基本信息和运营状态。

**门店信息**：

- 门店名称、地址、联系方式
- **门店类型**：
  - **直营门店**（direct）：
    - 所有权：平台直接所有和管理
    - 对外展示：统一展示为"某某店"（如：北京朝阳店）
    - 业务范围：完整的取还车服务、车辆管理、客服
    - **订单服务指引**：
      - 负责本门店订单的服务指引
      - 负责同一城市合作车辆订单的服务指引（优先级）
  - **加盟门店**（franchise）：
    - 所有权：加盟商所有，平台统一管理
    - 对外展示：统一展示为"某某店"（如：北京朝阳店）
    - 业务范围：完整的取还车服务、车辆管理、客服
    - **门店分润**：统一获得 10% 分润，归门店账户（与直营门店相同）
    - **订单服务指引**：
      - 负责本门店订单的服务指引
      - 负责同一城市合作车辆订单的服务指引（优先级）
  - **合作商户**（partner）：
    - 所有权：第三方合作方所有
    - 对外展示：标记为"合作门店"，特殊说明：取还车、后勤由第三方服务
    - 业务范围：主要为合作车辆的取还车服务
    - 收益分配：
      - 平台通过第三方订单平台完成差价获利（非平台服务费）
      - 本项目只记录订单发生及指引完成订单取还车流程
      - 无需计算任何分润
      - 订单管理内记录第三方平台实际订车费用，系统自动计算获利差价
    - **订单服务指引规则**：
      - 订单服务指引由同一城市的直营店或加盟店负责（优先级）
      - 如果合作门店所在城市没有直营店或加盟店，则由总部客服对接或总部分配客服对接
      - 服务等级：同一城市 > 总部逐级提升
- 门店状态：营业/停业/维护
- 门店照片和介绍

- **账户功能**：
  - 门店推广
  - 分润收益管理
  - 钱包提现（银行卡或微信）
  - 订单查看和管理
- **分润收益**：
  - 所有门店（直营和加盟）统一获得 10% 分润
  - 分润归门店账户，不归个人账户
  - 分润基于订单执行门店分配

**门店账户提现管理**：

- **提现信息报备**：
  - 钱包提现用银行卡或微信必须在总部报备
  - 修改提现信息需总部审批
  - 提现信息包括：银行卡号/微信账号、开户人姓名、开户行信息

- 门店账号不与 PLUS 会员叠加分润

- 门店账号的分润和员工个人账号的分润独立计算

**业务规则**：

- 门店信息完整准确
- 停业门店不显示在小程序端
- 门店地址用于用户定位匹配
- **合作商户只关联合作车辆，不参与平台分润**
- **直营门店和加盟门店统一获得 10% 分润，归门店账户**
- **所有门店（直营和加盟）的分润都归门店账户**（门店作为实体）
- **直营门店和加盟门店负责同一城市合作车辆订单的服务指引**
- **如果合作门店所在城市没有直营店或加盟店，由总部客服对接**

#### 5.3.5 门店特色服务管理

**功能描述**：配置门店独有的特色服务项目，用于该门店的服务和费用设置。**本配置为门店独立配置，每个门店可设置自己的特色服务。**

**业务定位**：

- **与增值费用管理的区别**：
  - 增值费用管理（4.9 节）：全局统一配置，所有门店共用的保险和附加服务，标准化产品
  - 门店特色服务：门店独有的服务项目，体现门店差异化优势
- **应用场景**：
  - 某些门店提供特色服务（如：免费洗车、免费代驾、特色路线推荐等）
  - 某些门店提供独特的增值服务（如：专属管家、高端餐饮、特色活动等）
  - 门店可根据自身资源和服务能力配置特色服务

**服务配置**：

- **基本信息**：
  - 服务名称、描述、分类、图标
  - 服务类型：免费服务/收费服务
  - 服务状态：启用/停用
- **价格设置**（收费服务）：
  - 服务价格、计费规则（按天/按次/一次性）
  - 会员价、优惠价设置
- **使用规则**：
  - 适用场景：房车租赁/营地预订/旅行定制
  - 数量限制：是否可选数量、最大数量
  - 推荐规则：是否在订单流程中推荐
  - 启用条件：是否需满足特定条件（如：租期 ≥3 天、特定车型等）
- **小程序端显示**：
  - 显示顺序、热门标识
  - 是否在门店详情页展示
  - 是否在订单流程中展示

**业务规则**：

- **门店独立性**：每个门店可独立配置自己的特色服务，不同门店的服务项目可以完全不同
- **服务可见性**：只有启用状态的特色服务才在小程序端显示
- **费用归属**：特色服务的费用归属门店，收入计入门店账户
- **与全局服务的关系**：
  - 全局附加服务（5.5 节）和门店特色服务在订单流程中同时展示
  - 用户可同时选择全局附加服务和门店特色服务
  - 门店特色服务优先显示（优先级高于全局服务）
- **配置同步**：门店经理可配置本门店的特色服务，配置后实时同步到小程序端

**小程序端对应**：

- 门店详情页：展示该门店的特色服务列表
- 订单流程中：在选择门店后，显示该门店的特色服务选项
- 订单确认页：显示已选择的门店特色服务及费用
- 订单详情页：展示已使用的门店特色服务

**配置权限**：

- 门店经理：可配置和管理本门店的特色服务
- 运营管理员：可查看所有门店的特色服务，可审核和编辑
- 超级管理员：拥有全部权限

**示例场景**：

  - 免费 24 小时道路救援（免费）
  - 专属房车管家服务（¥200/天）
  - 特色景点门票套餐（¥150/套）
- **门店 B 特色服务**：
  - 免费机场接送（免费）
  - 高端房车清洁服务（¥80/次）

#### 5.4.2 综合排序权重配置

**功能描述**：配置房车列表综合排序的权重系数。

**权重因子**：

| 因子名称       | 权重范围 | 默认权重 | 说明                 |
| -------------- | -------- | -------- | -------------------- |
| **房车评分**   | 0-100    | 30       | 用户评价的综合评分   |
| **预订热度**   | 0-100    | 25       | 近期预订次数和浏览量 |
| **价格合理性** | 0-100    | 20       | 价格与同类车型的对比 |
| **车辆新旧**   | 0-100    | 15       | 车龄和里程数         |
| **可用性**     | 0-100    | 10       | 可用日期数量         |

**计算公式**：

```
综合得分 = 房车评分×权重1 + 预订热度×权重2 + 价格合理性×权重3 + 车辆新旧×权重4 + 可用性×权重5
```

**配置方式**：

- 调整各因子权重（总和=100）
- 实时预览排序效果
- 一键恢复默认权重
- 保存后立即生效

**业务规则**：

- 所有权重之和必须等于 100
- 权重调整后影响所有房车列表
- 提供 AB 测试功能，对比不同权重效果
- 记录权重调整历史

**小程序端对应**：

- 房车列表默认按综合排序展示
- 排序结果根据配置的权重计算

**推荐规则**：

**配置项**：

**业务规则**：

### 5.10 分润管理功能

#### 5.10.1 分润基数配置

**可分润费用配置**：

| 产品类型   | 是否参与分润 | 分润基数   | 说明                 |
| ---------- | ------------ | ---------- | -------------------- |
| 房车日租金 | ☑️ 是        | 租金金额   | 核心收入来源         |
| 营地预订   | ☑️ 是        | 订单金额   | 营地服务费           |
| 旅行产品   | ☑️ 是        | 订单金额   | 定制旅游费用         |
| 增值服务费 | 根据配置     | 服务费金额 | 在增值费用管理中配置 |

**不可分润费用配置**：

在"增值费用管理"功能（4.9 章节）中，每个费用项都有"是否参与分润"勾选框：

- 勾选"☑️ 否"的费用：不计入分润基数，直接划归归属方
- 勾选"☑️ 是"的费用：计入分润基数，参与分润计算

**分润基数计算公式**：

```
步骤1：计算订单总额
订单总额 = 租金 + Σ(勾选"参与分润☑️是"的增值费用)

步骤2：扣除优惠
优惠后金额 = 订单总额 - 优惠券抵扣

步骤3：扣除不参与分润的费用
不参与分润费用 = Σ(勾选"参与分润☑️否"的费用)
可分润金额 = 优惠后金额 - 不参与分润费用

步骤4：各方分润
根据用户标签配置的分润比例计算各方收益
```

**优惠券处理规则**：

- 优惠券抵扣不影响分润基数计算
- 优惠成本由平台承担
- 分润基于优惠后的实际支付金额

#### 5.10.2 门店分润配置管理

**功能描述**：配置门店分润规则，管理门店收益分配和账户管理。

**门店分润规则**：

- **分润比例**：所有门店统一 10%（可配置）
- **适用门店**：
  - **直营门店**：获得 10% 分润，归门店账户
  - **加盟门店**：获得 10% 分润，归门店账户
  - **说明**：不区分直营门店或加盟门店，所有门店统一规则
- **分配规则**：
  - 基于订单的取还车门店（执行订单的门店）
  - 车辆在哪个门店执行订单，10% 门店分润就计入该门店
  - 统一规则，取消异地订单的特殊分配机制
- **归属账户**：
  - **所有门店（直营和加盟）的分润都归门店账户**（门店作为实体）
  - 门店账户与个人账户独立，不与 PLUS 会员叠加分润

**门店账户管理**：

- **账户类型**：门店账户（实体账户）
- **账户功能**：
  - 小程序端登录（用户名/密码登录模式）
  - 门店推广
  - 分润收益管理
  - 钱包提现（银行卡或微信）
- **账户管理规则**：
  - 门店统一在小程序端分配账号用于门店推广、登录等
  - 钱包提现用银行卡或微信必须在总部报备
  - 修改提现信息需总部审批
  - 门店账户与个人账户独立
- **账号独立性**：
  - 门店账号单独授予，不与 PLUS 会员叠加
  - 门店员工的工作账号如果单独购买 PLUS 会员，按正常分润规则获取推广收益
  - 门店账号的分润和员工个人账号的分润独立计算

**异地订单门店分润规则**：

- **取消特殊分配机制**：
  - 取消"后续异地门店订单：10% 原门店，90% 异地门店"的特殊分配规则
  - 统一规则：车辆在哪个门店执行订单，10% 门店分润就计入该门店
- **异地还车服务费归属**：
  - 不参与分润，作为费用归属直接分配给门店
  - 分配规则：50% 异地门店，50% 原门店
  - 归属方式：费用归属，不计入分润体系

**业务规则**：

- 所有门店（直营和加盟）统一获得 10% 分润
- 分润归门店账户，不归个人账户
- 门店账户提现信息需总部报备和审批
- 门店账号与个人账号独立，不叠加分润

#### 5.10.3 标签分润配置管理

**配置总览页面**：

显示所有参与分润的标签及其配置：

| 标签名称     | 标签类别 | 分润类型 | 分润比例           | 分润层级 | 操作   |
| ------------ | -------- | -------- | ------------------ | -------- | ------ |
| 众筹车主     | 众筹身份 | 车主分润 | 30%                | 一级     | [配置] |
| PLUS 会员    | 推广权益 | 推广分润 | 一级 5%，二级 1%   | 二级     | [配置] |
| 特殊推广用户 | 自定义   | 推广分润 | 自定义（最高 10%） | 一级     | [配置] |

**分润配置说明**：

- **门店分润**：所有门店（直营门店和加盟门店）统一获得 10% 分润，基于订单执行门店分配
  - 直营门店分润：归门店账户
  - 加盟门店分润：归门店账户
  - 不再使用"加盟商"或"企业分销商"标签用于分润
  - 门店账户与个人账户独立，不与 PLUS 会员叠加
- **PLUS 会员**：二级分销
  - 会员推广奖励：一级 20 元/单，二级 10 元/单（可配置）
  - 订单推广奖励：一级 5%，二级 1%（可配置）
  - 门店、加盟商统一通过购买 PLUS 会员获取推广收益
  - 门店员工如单独购买 PLUS 会员，按正常分润规则获取推广收益
- **众筹车主**：车主分润 30%（可配置），按份额比例分配
  - 支持虚拟车主"平台"，平台作为车主的分润单独计入车辆收益，不计入平台总收益
- **特殊推广用户**：自定义分润比例（最高 10%）
  - 从订单总额内分润
  - 分润比例可自定义，但最高不超过 10%

#### 5.10.4 分润叠加规则配置

- **不同类别的标签分润可叠加**：
  - 众筹车主分润（30%）+ 门店分润（10%）+ 推广分润（可叠加）
  - 例如：PLUS 会员一级推广（5%）+ PLUS 会员二级推广（1%）可同时存在
- **同一订单同标签只能获得一次分润**：
  - 同一用户不能因为同一标签获得多次分润
- **门店账号与个人账号独立**：
  - 门店账号单独授予，不与 PLUS 会员叠加分润
  - 门店员工如单独购买 PLUS 会员，按正常分润规则获取推广收益
  - 门店账号的分润和员工个人账号的分润独立计算
- **分润比例限制**：
  - 特殊推广用户最高 10%
  - 总分润比例建议设置预警机制（如总分润比例 > 80% 时预警）
  - 平台保留收益：确保平台剩余收益 ≥ 0（总分润比例 ≤ 100%）

#### 5.10.5 分润计算与记录

**触发时机**：订单状态变更为"已完成"

**分润计算时机**（修正）：

- **订单完成时立即计算**：
  - 订单状态变更为"已完成"时，立即计算该订单的分润
  - 分润记录保存到对应账户，状态为"待划拨"
  - 每个分润记录必须关联订单号
- **次月 5 日统一划拨**：
  - 每月 5 日统一将上月所有订单的分润划拨到用户账户
  - 生成订单清单，清晰展示每笔分润来源
  - 更新分润记录状态为"已划拨"

**计算流程**：

1. **订单完成时**：

   - 计算可分润金额（扣除不参与分润的费用）
   - 识别分润方（门店、车辆众筹车主、推广人）
   - 按配置比例计算各方分润
   - 生成分润明细记录（关联订单号，状态：待划拨）
   - 保存到用户账户或门店账户

2. **次月 5 日统一划拨**：
   - 系统自动汇总上月所有"待划拨"分润记录
   - 生成订单清单（每个用户/门店一份）
   - 批量划拨到用户钱包或门店钱包
   - 更新分润记录状态为"已划拨"
   - 发送分润通知和订单清单

**订单清单格式**：

- 每个用户/门店收到订单清单，包含：
  - 订单号（必填，关联具体订单）
  - 订单完成日期
  - 订单金额
  - 分润金额
  - 分润来源说明（如：众筹车主分润、门店分润、推广分润等）
- 清单支持导出和查询

**分润记录字段**：

- 订单号（必填，关联具体订单）
- 分润类型（众筹车主/门店分润/推广分润）
- 分润金额
- 计算时间（订单完成时间）
- 划拨时间（次月 5 日）
- 订单清单标识（用于生成清单）

#### 5.10.6 未分配分润统计

#### 5.10.8 业务规则

- **分润基数**：基于订单可分润金额计算
- **分润参与方**：
  - 门店：所有门店（直营和加盟）统一 10%，归门店账户
  - 众筹车主：30%，按份额比例分配
  - PLUS 会员：一级 5%，二级 1%（可叠加固定奖励）

  - 其他分润归个人账户
- **分润叠加**：
  - 不同类别的标签分润可叠加
  - 门店账号与个人账号独立，不叠加
- **订单清单**：每个分润记录必须关联订单号，划拨时生成清单
- 比例调整对新订单生效
- 分润记录永久保存

- IP 地址、设备信息、操作结果

**日志查询**：

- 按操作人员查询
- 按操作类型查询
- 按时间范围查询
- 按操作对象查询
- 按操作结果查询（成功/失败）

**日志分析**：

- 操作频率统计
- 异常操作识别
- 操作行为分析
- 安全事件分析

**日志存储与保护**：

- 日志永久保存，不可删除（超级管理员可归档）
- 日志加密存储，防止篡改
- 日志访问权限控制
- 定期备份日志数据

**业务规则**：

- 所有重要操作必须记录日志
- 日志记录实时写入，不可延迟
- 日志内容不可修改，确保真实性
- 敏感操作日志需特殊保护

### 7.4 员工管理功能

#### 7.4.1 功能概述

**业务目标**：管理系统用户账户（管理员、门店员工、客服等），统一管理员工信息、账户权限和状态。

**管理范围**：

- 管理员账户管理（总部管理员）
- 门店员工账户管理
- 客服人员账户管理
- 员工角色分配和权限管理
- 员工账户状态管理

**与用户管理的区别**：

- **用户管理（4.1 章节）**：管理 C 端小程序注册用户（租车用户）
- **员工管理（7.4 章节）**：管理系统内部员工账户（后台管理员、门店员工、客服等）

#### 7.4.2 管理员账户管理

**功能描述**：管理总部管理员账户的创建、编辑、禁用和角色分配。

**管理员类型**：

- **超级管理员**：系统最高权限管理员
- **运营管理员**：日常运营管理人员
- **财务管理员**：财务相关管理人员
- **客服管理员**：客服团队管理人员
- **内容管理员**：内容审核管理人员
- **数据分析师**：数据分析专员

**账户信息**：

- 基本信息：
  - 用户名、真实姓名、手机号、邮箱
  - 部门、职位、入职时间
  - 账户状态（正常/冻结/停用）
- 登录信息：
  - 登录账号（用户名/邮箱/手机号）
  - 密码设置（支持密码重置）
  - 最后登录时间、登录 IP、登录次数
- 权限信息：
  - 分配的角色（可多角色）
  - 功能权限列表
  - 数据权限范围

**账户管理操作**：

- **创建账户**：
  - 由超级管理员创建
  - 必填：用户名、真实姓名、手机号、角色
  - 初始密码：系统自动生成，首次登录需修改
  - 创建后发送账号和初始密码通知
- **编辑账户**：
  - 超级管理员可编辑所有信息
  - 运营管理员可编辑基本信息（姓名、手机、邮箱等）
  - 不能修改用户名（如需修改需删除重建）
- **禁用/启用账户**：
  - 冻结：临时禁用，保留数据，可恢复
  - 停用：永久禁用，数据归档，不可恢复
  - 禁用后立即生效，无法登录系统
- **重置密码**：
  - 管理员可重置任意员工密码
  - 重置后系统生成新密码并通知
  - 用户首次登录需修改密码

**业务规则**：

- 用户名唯一，不可重复
- 手机号和邮箱用于登录和找回密码
- 账户创建后必须分配至少一个角色
- 角色变更后权限立即生效
- 账户禁用前必须处理待办事项
- 所有操作记录操作日志

#### 7.4.3 门店员工账户管理

**功能描述**：管理门店员工账户的创建、编辑、禁用和门店关联。

**员工类型**：

- **门店经理**：门店管理人员，负责门店运营和员工管理
- **门店员工**：门店一线员工，负责订单取还车和客服接待

**账户信息**：

- 基本信息：
  - 用户名、真实姓名、手机号、邮箱
  - 所属门店、职位、入职时间
  - 账户状态（正常/冻结/停用）
- 门店信息：
  - 门店名称、门店 ID
  - 门店类型（直营/加盟/合作）
  - 门店地址
- 登录信息：
  - 登录账号（用户名/手机号）
  - 最后登录时间、登录次数
- 权限信息：
  - 分配的角色（门店经理/门店员工）
  - 数据权限范围（仅本门店数据）

**账户管理操作**：

- **创建账户**：
  - 由超级管理员或运营管理员创建
  - 必填：用户名、真实姓名、手机号、所属门店、角色
  - 初始密码：系统自动生成，首次登录需修改
  - 创建后发送账号和初始密码通知给门店经理
- **门店关联**：
  - 员工必须关联一个门店
  - 可查看和编辑所属门店
  - 门店经理可管理本门店员工账户（查看、编辑基本信息）
- **编辑账户**：
  - 超级管理员和运营管理员可编辑所有信息
  - 门店经理可编辑本门店员工基本信息
  - 门店员工仅可查看和修改个人信息
- **禁用/启用账户**：
  - 冻结：临时禁用，保留数据，可恢复
  - 停用：永久禁用，数据归档，不可恢复
  - 员工离职时必须停用账户
  - 账户停用后，关联的待办任务和订单需转交给其他员工

**门店员工管理功能**（门店经理视角）：

- 查看本门店所有员工列表
- 查看员工基本信息和工作状态
- 协助员工重置密码（需管理员审核）
- 查看员工工作统计（订单处理数、客服接待数等）

**业务规则**：

- 门店员工必须关联门店，不可跨门店
- 门店经理可查看和管理本门店所有员工
- 门店员工的数据权限仅限于本门店
- 账户停用前必须转交待办事项
- 所有操作记录操作日志

#### 7.4.4 客服人员账户管理

**功能描述**：管理客服人员账户的创建、编辑、禁用和客服组分配。

**客服类型**：

- **客服管理员**：客服团队管理人员，负责客服系统配置和团队管理
- **客服专员**：总部一线客服人员，负责全平台客户接待

**账户信息**：

- 基本信息：
  - 用户名、真实姓名、手机号、邮箱
  - 客服组、职位、入职时间
  - 账户状态（正常/冻结/停用）
- 客服信息：
  - 客服组名称、客服组 ID
  - 客服等级（初级/中级/高级）
  - 最大并发会话数
  - 服务评价统计
- 登录信息：
  - 登录账号（用户名/手机号）
  - 最后登录时间、在线状态
  - 今日接待会话数、总会话数
- 权限信息：
  - 分配的角色（客服管理员/客服专员）
  - 客服权限范围（全平台/指定门店）

**账户管理操作**：

- **创建账户**：
  - 由超级管理员或客服管理员创建
  - 必填：用户名、真实姓名、手机号、客服组、角色
  - 初始密码：系统自动生成，首次登录需修改
  - 创建后自动分配客服工作台权限
- **客服组分配**：
  - 客服必须分配到客服组
  - 可查看和编辑所属客服组
  - 客服管理员可管理本组客服账户
- **编辑账户**：
  - 超级管理员和客服管理员可编辑所有信息
  - 客服专员仅可查看和修改个人信息
- **禁用/启用账户**：
  - 冻结：临时禁用，保留数据，可恢复
  - 停用：永久禁用，数据归档，不可恢复
  - 账户停用后，进行中的会话需转交给其他客服

**客服工作统计**：

- 接待会话数统计（今日/本周/本月/总计）
- 平均响应时间
- 客户满意度评分
- 工单处理数统计
- 在线时长统计

**业务规则**：

- 客服人员必须分配客服组
- 客服管理员可查看和管理本组客服
- 客服专员的数据权限为全平台用户会话
- 账户停用前必须转交进行中的会话
- 所有操作记录操作日志

#### 7.4.5 员工角色与权限管理

**功能描述**：为员工分配角色，管理员工权限。

**角色分配**：

- **单角色分配**：一个员工分配一个角色（推荐）
- **多角色分配**：一个员工可分配多个角色（需谨慎）
- **角色变更**：可随时变更员工角色，权限立即生效

**权限继承**：

- 员工权限来自分配的角色
- 多角色时取权限并集（所有角色的权限合并）
- 权限变更后立即生效，无需重新登录

**权限管理**：

- 查看员工当前角色和权限
- 查看员工权限历史变更记录
- 批量修改员工角色和权限

**业务规则**：

- 角色分配由超级管理员或相应管理员执行
- 权限变更需记录日志
- 敏感权限变更需审批流程
- 权限最小化原则

#### 7.4.6 员工账户状态管理

**功能描述**：管理员工账户的状态，包括正常、冻结、停用等。

**账户状态**：

- **正常**：账户可用，可正常登录和使用系统
- **冻结**：临时禁用，保留数据，可恢复
  - 冻结原因：违规操作、安全风险、临时停职等
  - 冻结期限：可设置自动解冻时间
- **停用**：永久禁用，数据归档，不可恢复
  - 停用原因：离职、违规、账户注销等
  - 停用后账户不可恢复，数据归档保存

**状态变更流程**：

- **冻结账户**：
  - 操作人：超级管理员、运营管理员
  - 填写冻结原因和冻结期限（可选）
  - 冻结后立即生效，无法登录
  - 冻结期间保留所有数据
- **解冻账户**：
  - 操作人：超级管理员、运营管理员
  - 解冻后立即恢复登录权限
  - 解冻前可查看账户历史数据
- **停用账户**：
  - 操作人：超级管理员
  - 填写停用原因，必须确认
  - 停用前检查待办事项和关联数据
  - 停用后账户不可恢复
  - 数据归档保存，可查询但不可恢复账户

**业务规则**：

- 状态变更需记录详细日志
- 冻结账户保留所有数据和权限配置
- 停用账户前必须处理待办事项
- 停用账户不可恢复，操作需谨慎
- 所有状态变更需通知相关人员

#### 7.4.7 员工账户查询与统计

**功能描述**：查询员工账户信息，统计分析员工数据。

**查询功能**：

- **多维度筛选**：
  - 按角色：超级管理员、运营管理员、财务管理员等
  - 按部门：总部、门店、客服
  - 按门店：所属门店
  - 按状态：正常、冻结、停用
  - 按入职时间：自定义时间段
- **搜索功能**：
  - 用户名、真实姓名、手机号、邮箱
- **排序功能**：
  - 按创建时间、最后登录时间、登录次数

**统计功能**：

- **员工统计**：
  - 总员工数、各角色人数统计
  - 正常/冻结/停用账户统计
  - 各门店员工数统计
  - 客服组人数统计
- **活跃度统计**：
  - 今日/本周/本月登录人数
  - 活跃员工数统计
  - 长期未登录账户预警
- **权限统计**：
  - 各角色权限分布
  - 权限变更次数统计

**导出功能**：

- 导出员工列表（Excel）
- 导出员工统计报表
- 导出权限分配表

#### 7.4.8 业务规则

- **账户创建**：
  - 仅超级管理员和相应管理员可创建账户
  - 创建时必须分配至少一个角色
  - 初始密码系统生成，首次登录需修改
- **账户管理**：
  - 管理员只能管理权限范围内的员工
  - 门店经理只能管理本门店员工
  - 客服管理员只能管理本组客服
- **权限安全**：
  - 权限最小化原则
  - 敏感权限变更需审批
  - 权限变更记录日志
- **账户安全**：
  - 密码强度要求
  - 登录失败锁定机制
  - 异常登录检测和预警
  - 定期密码重置提醒
- **数据保护**：
  - 停用账户数据归档保存
  - 操作日志永久保存
  - 权限变更可追溯

#### 7.4.9 小程序端对应

**门店账户登录**（特殊说明）：

- 门店账户独立于员工账户
- 门店账户用于门店推广、分润管理等
- 门店账户支持用户名/密码登录
- 门店账户由总部统一创建和管理

**员工个人账户**：

- 门店员工如有个人小程序账号，可通过小程序端管理
- 员工个人账户与工作账户独立
- 员工可购买 PLUS 会员获取推广收益

### 7.5 智能预警与监控系统

#### 7.5.1 功能概述

**业务目标**：实时监控业务和系统状态，及时发现异常和风险，提前预警。

**管理范围**：

- 订单异常预警
- 车辆状态预警
- 财务异常预警
- 系统性能监控

#### 7.5.2 订单异常预警

**预警类型**：

- **订单支付异常**：支付失败率异常、支付金额异常、重复支付检测
- **订单状态异常**：长时间待支付订单、超时未取车订单、超时未还车订单
- **订单金额异常**：订单金额异常高或低、退款金额异常、优惠券使用异常

**预警规则配置**：

- 支付失败率阈值（如：单日失败率>10%）
- 订单超时时间阈值（如：待支付>30 分钟）
- 订单金额异常范围（如：<100 元或>50000 元）
- 退款率阈值（如：单日退款率>5%）

**预警处理**：

- 自动预警：系统自动检测并发送预警
- 预警通知：站内消息、邮件、短信通知相关人员
- 预警处理：运营人员查看预警并处理
- 预警记录：记录预警和处理结果

#### 7.5.3 车辆状态预警

**预警类型**：

- **车辆可用性预警**：某门店可用车辆不足（≤2 辆）、某车型可用车辆不足、车辆库存异常波动
- **车辆维护预警**：检查任务逾期预警、保养任务逾期预警、维修时间过长预警
- **车辆证件预警**：年检即将到期（提前 30 天）、保险即将到期（提前 30 天）、证件已过期预警

**预警规则配置**：

- 库存阈值配置
- 维护任务逾期时间阈值
- 证件到期提醒时间配置

**预警处理**：

- 自动标记异常车辆
- 通知相关人员（门店经理、运营人员）
- 记录预警和处理结果

#### 7.5.4 财务异常预警

**预警类型**：

- **收支异常**：单日收入异常波动、支出异常增长、收支不平衡预警
- **退款异常**：退款率异常高、大额退款预警、频繁退款用户预警
- **对账异常**：对账不平预警、支付渠道对账差异

**预警规则配置**：

- 收入波动阈值（如：±30%）
- 退款率阈值（如：>5%）
- 大额退款阈值（如：>10000 元）

**预警处理**：

- 自动生成预警报告
- 通知财务人员
- 启动调查流程

#### 7.5.5 系统性能监控

**监控指标**：

- **系统性能指标**：系统响应速度、服务可用性、业务处理效率
- **业务指标**：并发用户数、业务请求成功率、业务异常率

**监控告警**：

- 系统响应速度异常告警
- 服务不可用告警
- 业务异常率超阈值告警

**业务规则**：

- 监控数据实时采集
- 告警及时通知运维人员
- 监控数据用于业务运营决策

### 7.6 数据报表与 BI 分析

#### 7.6.1 功能概述

**业务目标**：提供强大的数据分析和报表功能，支持业务决策和运营优化。

**管理范围**：

- 自定义报表配置
- 数据钻取分析
- 趋势预测分析
- 对比分析

#### 7.6.2 自定义报表配置

**报表类型**：

- **业务报表**：订单报表（订单量、订单金额、订单趋势）、用户报表（用户增长、用户活跃、用户留存）、车辆报表（车辆使用率、车辆收益、车辆维护）
- **财务报表**：收支报表、分润报表、成本报表
- **运营报表**：门店运营报表、活动效果报表、客服服务报表

**报表配置**：

- **数据源选择**：选择数据表和字段
- **筛选条件配置**：时间范围、业务条件筛选
- **统计维度配置**：按时间、按地区、按产品等维度统计
- **图表类型选择**：表格、柱状图、折线图、饼图等

```
车型模板 (VehicleModel)

具体车辆 (Vehicle)
    ├─ vehicleModelId（关联车型模板，必填）

    ├─ originalStoreId（原始归属门店，用于收益分配）
    ├─ isRemoteTask（异地任务标识，临时标记）
    └─ partnerStoreId（合作车辆专用：合作方门店ID）
    ↓ (1对1，从车辆角度看)
门店 (Store)
    ├─ storeType（门店类型）
    │   ├─ direct：直营门店（对外展示：某某店）
    │   ├─ franchise：加盟门店（对外展示：某某店）
    │   └─ partner：合作商户（对外展示：合作门店）
    └─ displayName（对外展示名称）

    │ (当Vehicle.ownershipType='crowdfunding'时)
    ↓ (1对1)

    ├─ vehicleId（关联车辆，一对一）
    ├─ storeId（运营门店，车辆上线时指定）
    ├─ monthlyGuaranteedRevenue（月保底订单收入，可选配置）
    └─ status（项目状态）
    ↓ (1对多)
份额购买 (CrowdfundingShare)
    ├─ userId（购买用户，含平台账户）
    ├─ shareCount（份额数量）
    └─ purchaseAmount（购买金额）
    ↓ (多对1)
用户 (User)
    ├─ 标签：众筹车主（根据项目运营状态和份额持有情况）
    ├─ 标签：平台购车用户（平台购车参与众筹时添加）
    └─ 积分账户（OwnerPoints，随份额转让平移）
```

**关键关系说明**：

- **车辆 → 门店：1 对 1 关系**（每辆车唯一归属一个门店，storeId 必填）
- **门店 → 车辆：1 对多关系**（一个门店可以有多辆车）
- **车辆类型**：
  - 合作车辆（cooperative）：关联合作商户门店，不参与分润
  - 众筹车辆（crowdfunding）：关联直营/加盟门店，参与分润
- **门店类型**：
  - 直营门店/加盟门店：对外统一展示为"某某店"
  - 合作商户：对外展示为"合作门店"，特殊说明取还车、后勤由第三方服务
- **车辆 → 车型模板：多对 1 关系**（多辆车可以属于同一车型）
- **车辆 → 众筹项目：1 对 1 关系**（一个项目对应一台车辆）
- **运营门店指定时机**：车辆上线时指定，不是创建项目时指定
- **分润计算**：支持月保底订单收入（可选配置）

### C.4 分润计算规则（最终版）

**分润计算时机**：

- 订单完成时立即计算分润，次月 5 日统一划拨
- 每个分润记录必须关联订单号

**众筹车主分润**（单笔订单）：

```
单笔订单可分润金额 = 订单金额 - 不参与分润的费用

单笔订单运营成本 = 订单对应的（保险费 + 维护费 + 清洁费 + 平台服务费）

单笔订单净收益 = 单笔订单可分润金额 - 单笔订单运营成本

众筹车主单笔分润 = 单笔订单净收益 × 30% × 个人份额比例
```

**月保底收入处理**（月度汇总）：

```
如果配置了月保底订单收入：
  1. 每月1日统计上月所有订单收入总和
  2. 如果总订单收入 < 保底订单收入：
     - 计算保底差额 = 保底订单收入 - 总订单收入
     - 生成保底差额虚拟订单记录
     - 按保底差额计算额外分润，在次月5日一并划拨
  3. 如果总订单收入 ≥ 保底订单收入：按实际订单收入计算
```

**所有分润参与方计算**（单笔订单）：

```
订单可分润金额 = 订单金额 - 不参与分润的费用 - 优惠券抵扣

各方分润（基于单笔订单）：
- 众筹车主：订单可分润金额 × 30% × 份额比例
- 门店：订单可分润金额 × 10%（归门店账户，所有门店统一）
- PLUS会员一级推广：订单可分润金额 × 5% + 20元固定奖励
- PLUS会员二级推广：订单可分润金额 × 1% + 10元固定奖励
- 特殊推广用户：订单可分润金额 × 自定义比例（最高10%）
- 平台剩余收益：订单可分润金额 - 所有分润参与方的分润总额

平台总收益 = 平台剩余收益
平台作为车主的分润 = 单独计入车辆收益，不计入平台总收益
```

**分润划拨规则**：

- 订单完成时立即计算，分润记录状态为"待划拨"
- 次月 5 日统一划拨，生成订单清单
- 每个分润记录关联订单号

### C.5 合作车辆订单处理流程

```
订单创建（合作车辆）
    ├─ 记录订单基本信息
    ├─ 获取第三方平台订单信息
    ├─ 录入第三方平台实际订车费用
    └─ 系统自动计算获利差价（平台展示价格 - 第三方平台实际订车费用）
    ↓
订单服务指引分配
    ├─ 识别合作门店所在城市
    ├─ 查找同一城市的直营店或加盟店
    ├─ 如果有：分配给该门店负责指引（优先级1）
    └─ 如果没有：分配给总部客服或总部分配客服对接（优先级2）
    ↓
订单执行
    ├─ 指引门店/客服发送取车指引
    ├─ 用户在合作门店完成取车
    ├─ 指引门店/客服发送还车指引
    └─ 用户在合作门店完成还车
    ↓
订单完成
    ├─ 记录订单完成信息
    ├─ 计算并记录获利差价
    └─ 生成财务统计记录
    │
    └─ 注意：不参与任何分润计算
```

**关键规则说明**：

- **收益方式**：平台通过第三方订单平台完成差价获利（非平台服务费）
- **订单处理**：只记录订单发生及指引完成订单取还车流程
- **分润计算**：无需计算任何分润
- **差价计算**：获利差价 = 平台展示价格 - 第三方平台实际订车费用
- **服务指引**：同一城市 > 总部逐级提升
